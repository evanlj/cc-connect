# 2026-03-21 Squad 审核改为“用户裁决”机制

## 背景

在原有 Squad 流程中，Reviewer 会直接输出 `PASS/REWORK`，并由系统自动推进下一步或触发返工。  
实际使用中出现两个痛点：

1. 审核结论可能过度保守，触发“无意义阻塞”；
2. 用户无法在每轮审核后进行最终业务判断。

因此本次需求将“是否通过”的最终决策权交还给用户。

## 变更摘要

### 1) 新增审核裁决等待阶段

- 新阶段：`wait_review_judge`
- 每轮 Reviewer 输出后，流程暂停，等待用户执行裁决命令。

### 2) Reviewer 输出改为“建议型”

- Reviewer 不再输出 `PASS/REWORK` 最终结论；
- 仅输出结构化信息：

```json
{
  "review_result": "",
  "failed_reasons": ["..."],
  "suggestions": ["..."]
}
```

### 3) 新增命令

- `/squad judge-review <run_id> pass`
- `/squad judge-review <run_id> rework <原因与修改方案>`

行为说明：

- `pass`：推进到下一任务（仍遵守任务逐项确认门禁）；
- `rework`：将用户给出的“不通过原因与修改方案”注入执行者下一轮 Prompt，继续当前任务修订。

### 4) Checkpoint 记录增强

- 增加 `review_result` 字段；
- Reviewer 阶段先保存 `verdict=PENDING_USER`；
- 用户裁决后将 checkpoint 的 `verdict` 更新为 `PASS` 或 `REWORK`，用于最终报告回放。

## 影响与风险

### 正向影响

1. 用户可直接掌控是否通过，减少“审核卡死”；
2. 返工要求来源清晰（用户明确给出原因与方案）；
3. 审核结果与用户决策解耦，便于追溯。

### 风险点

1. 若用户长期不裁决，run 会停在 `wait_review_judge`；
2. 用户给出的返工要求过于模糊，可能导致执行者反复修改。

### 缓解

1. 在状态与任务详情中展示裁决命令；
2. 强制 `rework` 必须附带原因与修改方案；
3. 保留 `rework_count` 作为可观察指标（不再由系统自动硬失败）。

## 回滚说明

若需回滚到旧机制（Reviewer 自动裁决）：

1. 恢复 `core/squad_runner.go` 中 `parseReviewerVerdict` 自动分支；
2. 去除 `wait_review_judge` 阶段与 `/squad judge-review` 命令分支；
3. 将 Reviewer Prompt 恢复为输出 `PASS/REWORK`；
4. 重新构建并验证 `/squad approve-task` 后自动流转路径。
