# 2026-03-21 Squad 计划与任务双确认机制

## 1. 背景

在现有 Squad 流程中，用户只能在计划生成后执行一次 `/squad approve-plan`。  
实际使用中出现两个问题：

1. 计划内容（规则/任务拆解）没有在聊天窗口充分展示，用户难以快速核对；
2. 计划批准后会直接执行，用户无法对“当前即将执行的任务”再做逐项确认。

本次改造目标：**先看计划再批计划，再看任务再批任务**，把关键决策权放回用户。

---

## 2. 变更摘要

### 2.1 新增阶段与状态字段

文件：`core/squad.go`

- 新增阶段：`wait_task_approval`
- 新增运行字段：
  - `task_pending_id`：当前等待用户确认的任务 ID
  - `task_approved_id`：用户已确认可执行的任务 ID

### 2.2 计划展示增强（生成后即展示）

文件：`core/squad_runner.go`、`core/squad_parse.go`

- 计划生成后的通知消息中，新增计划预览（标题、摘要、任务列表、目标/验收摘要）
- 新增明确引导：
  - 查看完整计划：`/squad show-plan <run_id>`
  - 确认计划：`/squad approve-plan <run_id>`
  - 说明后续需要逐任务确认：`/squad approve-task <run_id> <task_id>`

### 2.3 新增“逐任务确认”门禁

文件：`core/squad_runner.go`

- 在执行循环中增加任务确认门禁：
  - 若当前任务未被确认，则切换到 `wait_task_approval`，并暂停执行；
  - 自动向用户推送当前任务详情和确认命令；
  - 仅当该任务被确认后才进入执行阶段。

### 2.4 新增 Squad 命令

文件：`core/engine_squad.go`

- `/squad show-plan <run_id>`：查看完整计划
- `/squad show-task <run_id> [task_id|task_index|current]`：查看任务详情
- `/squad approve-task <run_id> [task_id|task_index|current]`：确认当前待执行任务

并更新：

- `/squad start` 返回文案（明确“计划确认 + 任务逐项确认”）
- `/squad approve-plan` 返回文案（批准计划后提示下一步确认任务）
- `/squad status` 增加任务确认相关状态字段显示

---

## 3. 使用方式（新流程）

1) 启动  
`/squad start --repo <path> <需求>`

2) 查看计划  
`/squad show-plan <run_id>`

3) 确认计划  
`/squad approve-plan <run_id>`

4) 查看当前任务  
`/squad show-task <run_id> current`

5) 确认当前任务  
`/squad approve-task <run_id> <task_id>`

6) 任务执行完成后，系统会自动进入下一任务“等待确认”状态，重复 4)~5)。

---

## 4. 影响与风险

### 正向影响

- 用户可见性提升：计划与任务信息直接在会话中可读；
- 控制粒度提升：每个任务执行前都可人工把关；
- 风险更可控：避免计划批准后自动“连跑”到用户不希望执行的任务。

### 风险

- 流程比原先多一次确认动作，交互成本略增；
- 若用户忘记执行 `approve-task`，流程会停在等待阶段（这是有意的安全门禁）。
