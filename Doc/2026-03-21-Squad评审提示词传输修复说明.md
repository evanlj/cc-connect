# 2026-03-21 Squad评审提示词传输修复说明

## 1. 背景

在 `squad_20260321_122839_278400` 运行中，流程在 `reviewer ask` 阶段失败，报错：

- `ask status=500: The filename, directory name, or volume label syntax is incorrect.`

该问题表现为：

1. Planner、Executor 可正常执行；
2. Reviewer 在启动会话后极短时间内（几十毫秒）直接失败；
3. 失败发生在模型真正执行任务前，属于命令行传输层问题。

## 2. 变更摘要

### 2.1 代码修复

文件：`agent/codex/session.go`

- `Send()` 不再将完整 prompt 作为命令行参数传给 `codex`；
- 改为统一传入 `-`，并通过 `stdin` 传递 prompt 内容；
- 保持 `exec` 与 `exec resume` 两条路径一致：
  - 新会话：`codex exec ... --cd <workDir> -`
  - 续会话：`codex exec resume ... <threadID> -`
- 新增 `buildSendArgs()`，集中构建参数，降低参数分支重复。

### 2.2 测试补充

新增：`agent/codex/session_send_args_test.go`

- `TestBuildSendArgs_NonResumeUsesStdinPrompt`
- `TestBuildSendArgs_ResumeUsesSessionAndStdinPrompt`

覆盖点：

- 新会话与续会话均使用 `-` 作为 prompt 占位；
- 新会话包含 `--cd`，续会话不包含 `--cd`；
- 续会话正确携带 `threadID`。

## 3. 原因分析（结论）

Windows 下 `codex` 来自 npm shim（`codex.cmd`）。当 prompt 通过命令行参数传递且内容包含复杂符号（例如 JSON、引号、管道符组合）时，`cmd` 参数转发可能出现误解析，进而触发“文件名/目录名/卷标语法不正确”错误。

改为 `stdin` 传 prompt 后，避免了复杂字符串参与命令行解析，可稳定规避该类问题。

## 4. 影响与风险

### 正向影响

- 修复 Squad 中 Reviewer 阶段的传输层失败；
- 对 Planner / Executor / Reviewer 全角色生效；
- 降低 Windows 下复杂 prompt 的命令行解析风险。

### 风险评估

- 风险较低：`codex exec` 官方支持 `-` 从 stdin 读取 prompt；
- 行为变化点仅在 prompt 传输路径，不改变业务 prompt 内容与任务流程。

## 5. 验证证据

已执行：

1. 单元测试  
   - `go test ./agent/codex`
2. 全量测试  
   - `go test ./...`
3. 构建验证  
   - `go build -o cc-connect.exe ./cmd/cc-connect`

补充复现结论：

- 通过最小 Go 复现程序可稳定复现“命令行传 prompt 导致 exit status 255 + filename syntax”；
- 同等 prompt 改用 stdin 传输后不再触发该解析错误。
