# 多机器人讨论触发方式（mutilbot 版）

> 更新时间：2026-03-17  
> 适用仓库：`D:\ai-github\cc-connect`（分支：`mutil-bot`）  
> 适用场景：5 个机器人在同一个飞书群内围绕同一问题开展多轮讨论

---

## 1. 目前支持的触发入口

当前**正式触发入口**是飞书群内的 `/debate` 指令：

- 发起讨论：`/debate start ...`
- 查看进度：`/debate status [room_id]`
- 查看房间：`/debate list`
- 手动停止：`/debate stop <room_id>`

> 说明：当前没有单独的 `/debate` HTTP API；最稳定方式是直接在飞书群里触发。

---

## 2. 如何触发一场讨论（推荐流程）

## 2.1 前提

1. 5 个 bot 都已在同一个飞书群内。  
2. 5 个本地实例都已启动（`instance-mutilbot1~5`）。  
3. 建议由主持机器人（Jarvis 对应实例）来触发命令。

## 2.2 一条命令发起

在飞书群里发送：

```text
/debate start --preset tianji-five --rounds 3 --speaking-policy host-decide 请讨论：单机多机器人协作开发的最佳落地策略
```

发送后，系统会返回：

- 讨论 room_id
- 轮次上限
- 发言策略
- 后续可用命令（status/stop）

---

## 3. 触发命令模板（可直接复制）

## 3.1 最简触发（使用默认参数）

```text
/debate start 请讨论：如何把需求拆成可并行执行的任务
```

默认值：

- `preset = tianji-five`
- `rounds = 3`
- `speaking-policy = host-decide`

## 3.2 指定轮次/策略触发

```text
/debate start --rounds 4 --speaking-policy cover-all-by-end 请讨论：mutilbot 的稳定性优化清单
```

## 3.3 讨论中的控制命令

```text
/debate status
/debate list
/debate stop <room_id>
```

---

## 4. 发言策略说明（决定“本轮谁发言”）

- `host-decide`（默认）：主持人按轮次挑选发言角色（并非每轮 5 人都发言）
- `at-least-2`：每轮至少 2 个角色发言（人数不足时退化）
- `cover-all-by-end`：优先保证到最后一轮前覆盖全部角色

---

## 5. 当前默认角色映射（mutilbot 目录）

当仓库根目录存在 `mutilbot/` 时，`/debate` 默认使用以下映射：

| 角色 | 实例目录 | 默认项目名 |
|---|---|---|
| Jarvis | `mutilbot/instance-mutilbot1` | `instance-mutilbot1-project-jarvis` |

## 6. 一轮讨论触发后的判定标准（成功信号）

出现以下任一信息，说明触发成功：

1. 群内返回 `✅ 已创建并启动讨论：<room_id>`  
2. 看到 `【Jarvis】讨论开始（room=...）`  
3. 看到 `【Jarvis】第 N 轮开始，本轮发言：...`

---

## 7. 常见触发失败与排查

## 7.1 提示“debate 功能未启用”

检查实例 `data_dir` 是否可写、进程是否正常启动。

## 7.2 某角色不发言

优先检查该角色对应实例的 `api.sock` 是否存在：

```text
mutilbot/instance-mutilbotX/data/run/api.sock
```

## 7.3 讨论中途失败

执行：

```text
/debate status <room_id>
```

并查看 transcript 中是否出现 `ERROR:` 条目。

---

## 8. 建议的触发习惯（减少误操作）

1. 发起前先确认 5 个机器人都在线。  
2. 先用 2~3 轮小闭环验证，再拉长轮次。  
3. 每次讨论结束后记录 `room_id`，便于追踪和复盘。  
4. 长问题建议写成“目标 + 约束 + 期望产出”，避免跑题。

## 9. consensus 模式完整演示脚本（飞书可直接复制）

> 适用场景：需要“主持人先澄清 → 全员发散 → 多轮收敛 → 达成一致后总结”的讨论流程。  
> 建议：每条命令都带 `@Jarvis`，避免群路由丢失。

### 9.1 启动共识讨论

```text
/debate start --mode consensus --rounds 4 请讨论：如何把需求拆成可并行执行的任务 @Jarvis
```

预期：返回 `room_id`，随后主持人进入澄清阶段并提出问题。

### 9.2 用户回答澄清问题（关键）

将 `<room_id>` 替换为实际房间号：

```text
/debate answer <room_id> 目标：产出一份可执行并行开发计划；范围：需求拆分、依赖识别、owner分配、里程碑；非目标：不做UI原型与视觉设计；验收：输出任务清单（每项含owner+deadline+验收标准）并给出关键路径。 @Jarvis
```

预期：系统提示“已记录补充并恢复讨论”，随后进入主持人立论与全员发散。

### 9.3 查看流程状态

```text
/debate status <room_id> @Jarvis
```

关注字段：

- `mode: consensus`
- `phase: clarify_with_user / host_seed / all_diverge / host_collect / all_resolve / host_consensus_check / host_finalize`
- `iteration: N`

### 9.4 查看黑板

```text
/debate board <room_id> @Jarvis
```

重点看：

- `refined_topic`
- `open_questions`
- `consensus_points`
- `unresolved`
- `phase / iteration`

### 9.5 需要用户拍板时的补充（未收敛）

```text
/debate answer <room_id> 拍板：MVP先按A方案执行（先完成后端契约+主流程打通），B方案（复杂权限）下迭代处理；里程碑不变。 @Jarvis
```

如未自动继续，可手动：

```text
/debate continue <room_id> @Jarvis
```

### 9.6 结束或中止

```text
/debate stop <room_id> @Jarvis
```

@Jarvis
